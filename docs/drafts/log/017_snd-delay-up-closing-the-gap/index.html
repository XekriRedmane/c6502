snd_delay_up: closing the gap to hand-written

A small Apple II sound-effects routine. The hand-written 6502 emits speaker clicks with a descending pitch, busy-looping for an inner delay between each click. Nine instructions in the body, eight of them load-bearing. The C source is straightforward:

void snd_delay_up(uint8_t pitch __attribute__((reg("A"))),
                  uint8_t clicks __attribute__((reg("X"))))
{
    do {
        volatile uint8_t y __attribute__((reg("Y"))) = pitch;
        pitch = (uint8_t)(pitch + 1);
        while (--y != 0) { }
        (void)*sfx_click_ptr;
    } while (--clicks != 0);
}

Two parameters and a local, each pinned to a specific 6502 register via __attribute__((reg(".."))). The local y is volatile — the inner delay loop must execute, even though nothing observes its result. The volatile read through sfx_click_ptr triggers the speaker click (it points at $C030, the speaker toggle soft-switch on the Apple II).

The hand-written version of the same routine:

SND_DELAY_UP:
        TAY                             ; Y <- current pitch
        CLC
        ADC #$01                        ; A <- A + 1 (next pitch)
.inner:
        DEY                             ; inner delay
        BNE .inner
        CMP (ZP_SFX_CLICK),Y            ; speaker click; CMP preserves A
        DEX
        BNE SND_DELAY_UP
        RTS

Pitch lives in A across the entire loop. TAY copies the current pitch into Y for the inner delay; CLC+ADC #1 advances A to the next iteration's pitch. The inner loop is two instructions. The click uses CMP instead of LDA so A survives. Nine instructions total.

This is what we're trying to match. The starting point — what the compiler emitted before any of this work began — was twenty-something instructions and a soft-stack frame. We'll walk through the five optimizations that closed the gap to byte-identity in the body.

Where we started

The first cut, right after __attribute__((reg(".."))) landed end-to-end through the codegen, looked like this:

snd_delay_up:
   STA   __zpabi_snd_delay_up__pitch      ; entry stub for A param
   STX   __zpabi_snd_delay_up__clicks     ; entry stub for X param
   STA   __local_snd_delay_up__pitch      ; body copy of pitch
   STX   __local_snd_delay_up__clicks     ; body copy of clicks
   LDA   sfx_click_ptr                    ; DPTR staging
   STA   DPTR
   LDA   sfx_click_ptr+1
   STA   DPTR+1
.loop@0_start:
   LDY   __local_snd_delay_up__pitch      ; y = pitch
   INC   __local_snd_delay_up__pitch      ; pitch++
.loop@1_continue:
   LDA   __local_snd_delay_up__y          ; inner loop: --y != 0
   SEC
   SBC   #$01
   STA   __local_snd_delay_up__0
   LDA   __local_snd_delay_up__0
   STA   __local_snd_delay_up__y
   BNE   .loop@1_continue
   LDY   #$00
   LDA   (DPTR),Y                          ; volatile click
   DEC   __local_snd_delay_up__clicks
   BNE   .snd_delay_up@asm_ssa_split@0
   RTS
.snd_delay_up@asm_ssa_split@0:
   JMP   .loop@0_start

Twenty-three instructions in the body, plus the trampoline. The reg("Y") on the volatile local y wasn't honored — the local lived in a ZP slot, and --y compiled to a six-instruction read-modify-write through that slot. The reg("X")-passed clicks param lived in another ZP slot for the same reason. The reg attributes were recognized at the parser and the calling-convention level, but the body didn't take advantage of them.

Five optimizations later, the body would be ten instructions — identical to the hand-written version modulo the DPTR staging block. Each optimization was motivated by what was left over after the previous one, not designed up front.

The first peephole: tail-jump merging

The trampoline at the bottom of the function caught my eye first because it's structural, not register-allocation-dependent. The compiler emits

BNE .snd_delay_up@asm_ssa_split@0
RTS
.snd_delay_up@asm_ssa_split@0:
JMP .loop@0_start

where the hand-written has just

BNE SND_DELAY_UP
RTS

The shape Branch(cond, L); X; L: Jump(T) can always be rewritten to Branch(cond, T); X — branch directly to T instead of branching to a trampoline that jumps to T. When L has only the one reference (this Branch) AND the preceding instruction X is a control-flow terminator (Return / Jump / another Ret), the trampoline itself becomes dead and can be dropped along with its Label.

The "terminator before the Label" check matters. Without it, the fall-through path from above the Label would have flowed into the Jump, and dropping the Jump would change semantics. RTS terminates, so in our case the trampoline is unreachable after the retarget.

A first attempt dropped the trampoline whenever the Label's reference count went to zero, with no terminator check. That broke forty-five tests across the chapter corpus — functions where the Label sat between two reachable basic blocks, and the "orphan" Jump was actually reached by fall-through from above. The terminator gate killed the regressions without losing the snd_delay_up win.

The inner loop: DEY for register-pinned volatile counters

The biggest cycle savings hide in the inner delay. The compiler emits six instructions per iteration for what should be two:

compiler (before)
LDA __local_..._y
SEC
SBC #$01
STA __local_..._0
LDA __local_..._0
STA __local_..._y
BNE .loop@1_continue
hand-written
DEY
BNE .inner

The compiler's six-instruction sequence is the natural lowering of --y when y is volatile-pinned to Y: read y (volatile), compute y-1, write y (volatile, routed through a temp by SSA destruction). The existing dec_peephole folds single-byte SBC-#1 chains on stable memory to DEC, but it conservatively skips register operands when the volatile bit is set — partly because volatile semantics generally require explicit memory accesses.

For a register-pinned volatile, that defensiveness is too strong. The C standard requires each volatile access to execute "as if" it were a memory access. When the storage IS a register, the register access IS the access. DEY performs a single read-modify- write of Y in one instruction — exactly one volatile-observable read plus one volatile-observable write, matching what --y demands.

The new dec_register_peephole recognizes the six- instruction shape Mov(R, A); SEC; Sub(Imm(1), A); Mov(A, tmp); Mov(tmp, R); Branch(NE|EQ, _) when R is X or Y, and folds it to Dec(R); Branch(NE|EQ, _). The soundness gates are:

  • R is X or Y (DEX or DEY).
  • A is dead at both successors of the Branch — DEC R doesn't touch A, so the original sequence's A=y-1 result must be unobserved.
  • The spill temp is dead after the rewrite — its only role was to route the value through memory.

After this lands, the inner loop drops from 17 bytes to 5 bytes, from a measured ~14 cycles per iteration to 5. For a function called with pitch=128, that's the difference between 1800 cycles and 640 cycles in the inner loop alone.

The volatile click: CMP for discarded reads

The next thing standing in the way of A-pinning is the volatile click. The compiler emits LDA (DPTR),Y for (void)*sfx_click_ptr. The LDA performs the volatile read, but it also clobbers A — destroying whatever value pitch had been kept in.

The hand-written version uses CMP (DPTR),Y instead. CMP performs the same memory read (same volatile semantics), but only sets flags — A is unchanged. When the loaded value is discarded (as it is here, since the C code is (void)*ptr) AND the flags from the load aren't observed downstream, CMP and LDA are interchangeable for the read's side effect.

The new volatile_void_read_cmp peephole matches

Mov(<indirect>, Reg(A), is_volatile=True)

and rewrites to Compare(Reg(A), <indirect>) when A is dead after the Mov and the next instruction overwrites N/Z (so the Compare's flag effect is unobservable). The flag check matters because LDA sets N/Z based on the loaded byte while CMP sets them based on A - byte — different values when A and the loaded byte differ. The next instruction in our case is DEX, which clobbers N/Z; the rewrite is safe.

This rewrite alone is a small win (A is preserved across the click) but it unlocks the next bigger optimization.

The big one: loop-carried scalar promotion to A

With CMP-for-discarded-read in place, A survives the click. With DEY; BNE in the inner loop, A isn't touched in the inner delay either. The body, after the two prior optimizations, looks like this:

snd_delay_up:
   STA __local_..._pitch         ; save A to local
   ; DPTR staging clobbers A here
.loop@0_start:
   LDY __local_..._pitch         ; reload pitch into Y
   INC __local_..._pitch         ; pitch++
.loop@1_continue:
   DEY
   BNE .loop@1_continue
   LDY #$00
   CMP (DPTR),Y
   DEX
   BNE .loop@0_start
   RTS

The local __local_..._pitch serves only as a "pitch register" — written at function entry from A, read at the top of each iteration into Y, then incremented in place. Nothing outside the function observes it, and nothing inside the function reads it besides those two sites. If A is free in the loop body (and it now is), the local can be promoted to A:

  • Entry STA: replace with TAY — park pitch in Y while the DPTR staging clobbers A.
  • End of preamble: insert TYA — restore pitch to A just before the loop starts.
  • In-loop LDY M: replace with TAY (Y = pitch).
  • In-loop INC M: replace with CLC; ADC #$01 (A = pitch + 1, ready to carry across the loop edge into the next iteration's TAY).
  • The local disappears entirely.

This is scalar promotion of a loop-carried memory cell to a register — the same family as loop_counter_to_x (which does it for X-pinned counters), but for A and with a specific trigger pattern (entry STA + in-loop LDY/INC).

The new promote_loop_carried_to_a pass walks each function looking for this exact shape. The eligibility gates are strict: exactly one entry STA from Reg(A) to a __local_<fn>__* slot, exactly one LDY M; Inc(M) pair, the slot has no other access sites, Y is unused across the preamble (so the parked value survives), and A is unwritten between the in-loop INC site and the back-edge Branch (allowing only CMP-with-dead-flags as an A-read in that region).

The pass also has to deal with the SSA name layering. A Pseudo's name at this point may carry two stacked suffixes: @1.pitch becomes @1.pitch.2 (TAC-SSA's def counter) and then @1.pitch.2.b0.v3 (asm-SSA's byte+version suffix). Recovering the c99-level base name requires stripping both. The first attempt at the SSA-name-matching helper stripped only the asm-SSA suffix; pins for the resulting @1.pitch.2-rooted names never matched the @1.pitch-keyed pin map, and the pass silently did nothing.

The last byte: dropping a redundant LDY

After the four prior optimizations, one small wart remained:

.loop@1_continue:
   DEY
   BNE .loop@1_continue
   LDY #$00                       ← redundant, Y already 0
   CMP (DPTR),Y

The DEY; BNE fall-through has Y == 0 by definition (the BNE wouldn't have fallen through otherwise). The LDY #$00 sets Y to a value it already holds. But the LDY isn't in the asm IR as a separate atom — it's emitted by asm_emit as setup for the Indirect(0) operand on the Compare. The IR is just Compare(Reg(A), Indirect(0)).

The fix isn't an IR-level peephole on the LDY (it doesn't exist yet) but an operand rewrite. The asm IR has two indirect-Y variants: Indirect(offset) emits LDY #offset; <op> (DPTR),Y (forces Y to the offset), and IndirectY() emits just <op> (DPTR),Y (uses Y as-is). When we know Y is already 0 at the access point, rewriting Indirect(0) to IndirectY() drops the emit-time LDY.

The new dec_branch_indirect_y_fold peephole matches the specific three-instruction adjacency Dec(Reg(Y)); Branch(NE, _); <op with Indirect(0)> and rewrites the Indirect(0) to IndirectY(). Two bytes and two cycles per call, the smallest of the wins but the one that finally makes the body byte-identical.

Where we landed

After all five optimizations:

compiler output (body)
.loop@0_start:
   TAY
   CLC
   ADC #$01
.loop@1_continue:
   DEY
   BNE .loop@1_continue
   CMP (DPTR),Y
   DEX
   BNE .loop@0_start
   RTS
hand-written
        TAY
        CLC
        ADC #$01
.inner:
        DEY
        BNE .inner
        CMP (ZP_SFX_CLICK),Y
        DEX
        BNE SND_DELAY_UP
        RTS

Identical opcodes, identical addressing modes, identical cycle counts. The only delta is the indirect pointer: the compiler uses DPTR (the runtime's scratch indirect-pointer pair), the hand-written uses ZP_SFX_CLICK (a fixed ZP location holding the pointer to $C030).

The compiler still emits a six-instruction DPTR staging block at the top of the function:

TAY                          ; park pitch in Y across staging
LDA sfx_click_ptr
STA DPTR
LDA sfx_click_ptr+1
STA DPTR+1
TYA                          ; restore pitch to A

That's the remaining cost. The hand-written sidesteps it by arranging for ZP_SFX_CLICK to be set up by the input handler when the speaker is muted/unmuted — no per-call staging needed. To match that in C source, we'd need an attribute like __attribute__((zp)) extern const volatile uint8_t * sfx_click_ptr; telling the compiler the pointer is ZP-resident, and a corresponding lowering that emits (sfx_click_ptr),Y directly instead of staging through DPTR. That's deferred.

What the journey teaches

Each optimization was motivated by what was left after the previous one. None of them were designed up front. The order matters: CMP-for-discarded-read had to land before loop-carried-promotion would have a clean body to promote into. dec_register_peephole had to land before promote_loop_carried_to_a would see an A-clean inner loop. The order fell out naturally from looking at the intermediate output and asking "what's the next thing that doesn't match hand-written?"

The eligibility gates on the new passes are strict — every one of them is a structural pattern match with explicit constraints ("R must be X or Y", "A must be dead at both successors", "Y must be unused across the preamble"). General loop-aware analyses would catch more cases, but the cost is a much bigger pass with more correctness surface. The structural matches caught the snd_delay_up case without disturbing any of the existing 2700+ tests.

The most interesting failure along the way was a generalization attempt I had to back out. Extending passes/asm_aliasing.may_alias with a rule that "indirect-Y reads don't alias Data(__zpabi_*) / Data(__local_*) slot symbols" looked clean — the slots are compiler-synthesized ZP, the user can't construct their addresses, no indirect read should reach them. It broke forty-five tests. Address-taken locals get routed to the same pool by the call-graph-disjoint allocator: &local gives a pointer to a pool byte, and an indirect-Y read through that pointer DOES alias the local's slot. The general rule was unsound; __zpabi_* slots specifically aren't address-takable in C (no syntax for &__zpabi_*; ¶m on a reg-attributed param is a type-check error), so the narrow rule held. The targeted dead_reg_entry_stub_drop uses that narrow invariant without touching the general aliasing rule.

"Generalize what works, don't generalize what's narrow" is the takeaway. The narrow rule is true. The broad rule isn't.
